## HTML 生成规则

- 单文件、自包含、UTF-8 编码，不依赖外部 CDN / 远程 JS / 远程 CSS，所有样式内联；
- 支持按严重程度、状态、证据等级筛选，问题卡片支持折叠/展开，带目录导航；
- 每个问题卡片明确标注 ID、状态、严重程度、证据等级、复现状态；
- HTML 不允许作为唯一事实来源；如需更新问题状态，应重新运行 qa-check 读取 JSON 生成新版本。

## 条件章节输出规则

- `## 核查模式与资料完整度`：所有模式必须输出。
- `## 最小需求口径`：**仅 Hotfix 无文档核查模式输出**，紧跟在「核查模式与资料完整度」之后追加；完整提测核查、无用例核查、无 PRD 核查模式严禁输出此节。Hotfix 无文档核查模式时追加的模板如下：
  ```
  ## 最小需求口径
  - **本次要解决的问题**：
  - **用户预期结果**：
  - **已知影响范围**：
  - **明确不应受影响的功能**：
  - **资料来源**：
  ```
- 无 PRD 核查模式下有 bug 单、工单、MR 描述等资料时，只在「核查模式与资料完整度」表格中说明资料来源，不单独输出「最小需求口径」节。
- 最终报告不得包含 HTML 注释、模板注释或任何"删除本节"之类的编辑提示。

## 报告内容格式（Markdown 模板）

---

# <业务功能>-提测核查初版报告

分支：`<提测分支>` → `<基准分支>`
变更统计：X 个文件，+X -X 行

## 核查模式与资料完整度

| 项目 | 状态 |
| --- | --- |
| 报告阶段 | Draft / 初版 |
| 核查模式 | 完整提测核查 / 无用例核查 / 无 PRD 核查 / Hotfix 无文档核查 |
| PRD | 已提供 / 未提供 / 无权限 / 由 bug 单或变更说明替代 |
| 测试用例 | 已提供 / 未提供 |
| 变更说明 | 已提供 / 根据 commit 和 diff 推断 / 用户补充确认 |
| 结论边界 | 本次可判断：...；本次不能判断：... |

## 报告阶段与结论边界

- **当前阶段**：Draft / 初版
- **是否已提供测试入口**：是 / 否
- **是否已进行实测**：否
- **本报告可判断**：代码证据、理论风险、测试盲区、建议验证场景
- **本报告不可判断**：真实环境是否已复现、线上表现是否完全一致、最终验收结论

## 实测资料需求清单

如缺少以下资料，在生成终版报告前需用户提供：

| 资料类型 | 是否已提供 | 说明 |
| --- | --- | --- |
| 测试环境入口（URL / 包 / 小程序） | 否 | |
| 测试账号 / 角色 / 登录方式 | 否 | |
| 测试数据 / 构造方式 | 否 | |
| 环境配置 / 开关状态 | 否 / 不需要 | |
| 特殊前置状态 | 否 / 不需要 | |

## 架构说明（影响测试策略）

当功能涉及多仓、多端、web-view、H5、小程序或服务端时必须写，说明各功能模块的实际运行位置和为什么测试策略会受影响。

| 功能模块 | 实际运行位置 | 备注 |
| --- | --- | --- |
| ... | ... | ... |

## 一、发现的问题（代码直接验证）

从 diff 或源文件中直接读到的确定性问题，开发需要修复。

**问题 N：[简短标题]**
- **现象**：描述用户/测试会观察到的异常表现
- **复现步骤**：
  1. ...
  2. ...
  3. ...
- **原因**：基于代码路径说明为什么会发生
- **代码位置**：`仓库/文件路径` 中的 `函数名/逻辑块`

如无发现问题，写「未发现代码可直接验证的问题」。

## 二、疑似问题 / 需实测确认

包含：代码推断但缺实测验证的问题、平台兼容/环境差异、跨仓依赖契约、黑盒测试主流程不容易覆盖的边界分支、架构推断风险、**状态依赖类风险（续课/恢复/跳转/免登后某环节副作用未恢复，导致状态/集合/高亮/进度不一致——来自 Step 2.6 矩阵中未确证为 bug 的行）**。**不是代码 bug，开发无需修改，但测试必须验证。**

**疑似问题 N：[简短标题]**
- **问题来源**：本次新增 / 本次修改引入 / 旁路改动 / 既有风险被本次场景暴露 / 资料不足
- **可能现象**：测试可能观察到的表现（不得写"已复现"）
- **推断依据**：代码路径 / PRD / 测试用例 / 平台机制 / 跨仓依赖
- **理论复现步骤（待验证）**：
  1. ...
- **验证观察点**：实测时重点关注的现象或指标
- **需要的测试资料**：账号 / 数据 / 平台 / 特殊状态
- **无法实测原因**（如适用）：
- **对应用例**：用例编号/名称；没有则写「无直接对应」

如无疑似问题，写「未发现需要额外实测确认的疑似问题」。

## 三、风险提示

- **数据库变更**：有/无，并说明依据
- **接口变更**：有/无，说明新增/修改/依赖方/兼容风险
- **跨仓库依赖**：有/无，说明前后端、H5/小程序、服务间依赖
- **配置/开关变更**：如有，说明 Apollo、环境变量、灰度开关等影响
- **缓存/异步/消息风险**：如有，说明对测试回归的影响
- **权限/登录态/数据状态风险**：如有，说明需要覆盖的账号或数据状态

## 四、建议回归场景

只列有 diff 证据影响**已有功能**的场景（旧功能被本次改动波及），无依据的不写。

**场景 N：[功能名]**
- **原因**：哪个文件/函数被修改，以及为什么会影响已有功能
- **建议操作**：具体测试路径和观察点

如无受影响的已有功能，写「本次改动未发现影响已有功能的回归场景」。

## 五、建议补充测试场景

列**新功能**里代码和架构风险推导出的黑盒盲区，补足黑盒测试容易遗漏的问题。不以审查测试用例完整性为目的。

**场景 N：[功能名]**
- **补充原因**：边界分支、状态/缓存/异步、跨端跨仓、平台环境差异、失败链路或权限登录态风险、**数据质量（脏数据/短语/缩写/无效词/空值，来自 Step 3.6 第 3 类）**、**状态依赖（绕过流程后副作用未恢复，来自 Step 2.6）**
- **相关依据**：PRD/用例/代码路径/架构事实
- **建议操作**：具体测试路径和观察点

> 数据质量类场景即使代码静态查不出对错也必须列出，提示用真实数据验证内容质量。

## 特别说明

**明确排除 / 暂缓测试**
- PRD、用户或测试环境明确声明本次不测、暂缓、无法测试的内容

**资料或权限不足导致未覆盖**
- 缺仓库、缺分支、缺 PRD 权限、缺测试用例、缺环境、缺数据等导致未覆盖的内容

## 注意事项

- 如果 PRD 文档无法通过工具读取，提示用户将 PRD 关键功能点直接粘贴进来
- 如果用户没有提供测试用例，跳过用例覆盖分析和对应用例匹配；仍必须基于 diff、改动影响范围和用户补充说明输出建议回归场景和建议补充测试场景
- 如果分支不存在或 git 命令报错，及时告知用户并引导排查
- 如果用户只有 HTML、没有 JSON 或 Markdown，可以尽量从 HTML 提取问题清单，但必须提示「HTML 不是推荐的结构化来源，可能无法完整恢复状态字段」；建议用户在下次核查后保留 JSON 文件

## 复测报告模板

复测报告写入：`~/Desktop/qa-check-reports/<需求名称>/<日期>-<分支名>-<shortCommit>/recheck/`，同时生成 `qa-check-recheck-<日期>.md`、`.json`、`.html` 三类文件，不覆盖 `final/` 目录中的上一轮报告。

# <业务功能>-复测报告

分支：`<当前提测分支>` → `<基准分支>`
