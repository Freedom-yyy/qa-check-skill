# /qa-check Skill 规范

## 目标

`/qa-check` 用于开发提测后，基于需求文档、测试用例和代码变更扫描开发代码，提前发现代码中的确定问题、疑似问题，以及黑盒测试容易遗漏的风险场景。

第一交付物必须是一份业务语气自然的提测核查报告。报告不输出"是否验收合格""是否允许提测""是否通过验收"等决策性结论，只输出问题、风险、回归建议和补充测试建议。

## 非目标

- 不替代测试验收结论。
- 不以审查测试用例完整性为主要目的。
- 不把所有疑点都写成确定 bug。
- 不为了动态验证阻塞第一份报告交付。
- 不输出带有明显工具标签的标题，例如"AI 扫描报告""黑盒盲区报告"。

## 输入

至少收集以下信息：

1. 代码仓库路径（本地绝对路径）。
2. 提测分支。
3. 对比基准分支，默认 `master`。
4. 需求文档链接，优先支持飞书文档（支持 `/docx/`、`/docs/`、`/wiki/` 格式）。
5. 测试用例，可为空；支持 XMind（自动检测 content.xml / content.json 格式）、本地文件、表格、纯文本或用户粘贴内容。

**多仓库场景**：如涉及多仓库、多端或 web-view/H5/小程序/服务端联动，应明确每个仓库的本地路径、分支和职责。多仓库并行分析，最终合并为一份报告，按功能模块组织内容（标注代码所在仓库），不按仓库分章节。

## 推荐流程

### 1. 收集基本信息

向用户一次性询问：仓库路径（可多个）、提测分支、基准分支、PRD 链接、测试用例。不逐条追问。

### 2. 识别项目类型（自动探查 + 人工确认）

收到仓库路径后，先自动探查根目录结构和关键配置文件（`package.json`、`go.mod`、`pom.xml`、`pages.json`、`app.json`、`AndroidManifest.xml`、`Podfile` 等），判断：

- 项目类型（前端 Web / 小程序 / 原生 App / 后端服务 / Flutter 等）
- 核心目录（将对这些目录的文件全文阅读）
- 跨层/跨仓依赖（如 web-view 加载 H5、后端调用外部服务等）

探查完成后向用户确认，**等用户回复后再继续**。用户可纠正项目类型和核心目录。

### 3. 获取代码差异

多仓库并行执行：

```bash
git fetch origin
git diff origin/<base>...origin/<qa-branch> --stat
git diff origin/<base>...origin/<qa-branch> --name-only
git log origin/<base>..origin/<qa-branch> --oneline
```

若使用本地分支，则去掉 `origin/` 前缀。

**核心文件处理**：对落在确认的核心目录内的文件，使用 `git show origin/<提测分支>:<文件路径>` 读取完整源码，不用 grep 提取——grep 只能找到预设关键词，全文阅读才能发现预设之外的问题。非核心文件（第三方依赖、构建产物、自动生成文件）只保留 `--stat` 摘要。

### 4. 阅读 PRD 和测试用例（并行）

PRD 主要用于理解需求目标、业务规则、明确排除范围和测试环境限制。读完后立即提取：

- PRD 中明确标注为"重点/测试重点"的功能（后续分析必须逐一核查）
- PRD 中标注为"暂不支持/测试环境无法测试"的功能（进入特别说明章节）

测试用例主要用于理解测试关注路径。只有当用例和代码风险判断相关时，才指出用例覆盖差异。

### 5. 架构 × 用例交叉分析

PRD、用例、代码都读完后，逐条扫描测试用例，判断每条用例依赖的能力在当前运行环境下是否有已知限制。重点关注：

- 音视频自动播放（autoplay）
- 系统返回 / 导航拦截（back button、手势返回）
- 文件上传 / 下载
- 剪贴板读写
- 推送通知 / 后台保活
- 摄像头 / 麦克风 / 定位等系统权限
- 本地存储（容量限制、隔离策略）
- 网络请求（跨域、证书限制）

有已知限制的记录为「架构推断风险」，进入报告的「疑似问题/需实测确认」章节。此步骤不依赖 diff，只需要：用例说明了什么 + 当前环境的已知约束。

### 6. 综合访谈（**必须执行，不可跳过**）

完成上述所有步骤后，在输出报告之前，使用 **AskUserQuestion 工具**将以下三类问题结构化呈现，一次性向用户确认：

> **重要**：即使用户在初始 prompt 中说了"直接执行""直接出报告"等字样，也必须先完成本步骤。"直接执行"的含义是省去信息收集的重复确认，不是跳过分析流程中的步骤。

**一、范围对齐（每次必问）**

- 审查模式：全面扫描 还是 有明确重点区域？
- 已在其他平台/阶段验证过的功能，本次可跳过？
- 测试环境限制、不在本次需求内的功能？
- 你自己已有哪些顾虑或希望重点关注的地方？

**二、架构追问（根据发现按需触发，不相关的跳过）**

| 发现的特征 | 追问内容 |
|-----------|---------|
| 存在 web-view 加载 H5 | H5 仓库是否也在本次提测范围？ |
| 多平台构建（微信/抖音/iOS/Android） | 哪些平台需要覆盖，哪些暂不测试？ |
| 存在数据库 migration | 是否已通知 DBA 审核？是否需要回滚方案？ |
| 新增或修改对外 API | 调用方是否已同步？是否需要版本兼容？ |
| 涉及第三方服务 | 测试环境的第三方服务是否可用？ |
| 存在配置开关 / Feature Flag | 测试环境开关状态是否已确认？ |

**三、疑点确认（根据分析结果按需触发）**

- 代码中有改动、但 PRD 未提及的逻辑：逐条描述，询问是否属于本次需求范围
- PRD 提及、但 diff 中未找到对应改动的功能点：询问是否在其他仓库或已提前上线

问题总数控制在 5 题以内；超出时合并低优先级问题。用 AskUserQuestion 工具的多选选项减少用户手打文字的负担。

用户回答后，将所有答案作为最终约束带入报告生成。

### 7. 扫描代码问题和黑盒易遗漏场景

重点扫描：

- 主流程之外的边界分支。
- 状态、缓存、异步、重复操作。
- 跨端、跨仓、跨服务依赖。
- 历史功能被新逻辑间接影响。
- 平台、环境、账号数据差异。
- 失败链路和异常恢复。
- 权限、登录态、数据隔离。
- 接口、配置、消息队列、定时任务、埋点等黑盒不易直接观察的变更。

### 8. 质量自检（输出报告前，结果只在对话展示）

报告内容生成完毕后，对以下 8 条标准逐一自检。自检结果只在对话中告知用户，不写入最终文档：

1. 是否避免了"验收通过/不通过"的决策性结论。
2. 是否把确定问题和疑似问题分开。
3. 每个确定问题是否有代码位置和原因。
4. 每个疑似问题是否有详细推断依据和建议验证方式。
5. 风险提示是否和本次 diff 有关。
6. 建议回归场景是否有 diff 证据。
7. 建议补充测试场景是否聚焦黑盒容易漏的代码/架构风险。
8. 特别说明是否区分了明确排除和资料/权限不足。

有不达标项则修正后再输出。

### 9. 产出报告并写入文档

报告使用业务功能命名：`# <业务功能>-提测核查报告`

**写入优先级**：
1. 优先使用飞书 MCP 的 `docx_builtin_import` 写入飞书云文档，文件名：`<业务功能>-提测核查报告-<日期>`，成功后告知文档链接。
2. 若飞书写入失败，回退写入本地文件：`~/Desktop/qa-check-<提测分支名>-<日期>.md`，告知文件路径。

## 报告模板

```markdown
# <业务功能>-提测核查报告

分支：<qa-branch> → <base>
变更统计：<仓库/模块/文件数/行数摘要>

## 架构说明（影响测试策略）

当功能涉及多仓、多端、web-view、H5、小程序或服务端时必须写。
说明实际运行位置，以及为什么测试策略会受影响。

| 功能模块 | 实际运行位置 | 备注 |
| --- | --- | --- |
| ... | ... | ... |

## 一、发现的问题（代码直接验证）

**问题 1：<标题>**
- **现象**：<用户或测试可能观察到的异常表现>
- **预期结果**：<可选；仅当有助于说明问题时填写>
- **复现步骤**：
  1. ...
  2. ...
  3. ...
- **原因**：<基于代码路径说明为什么会发生>
- **代码位置**：<仓库/文件/函数或逻辑块>

如无发现问题，写"未发现代码可直接验证的问题"。

## 二、疑似问题 / 需实测确认

包含：代码推断但缺实测验证的问题、平台兼容/环境差异、跨仓依赖契约、
黑盒测试主流程不容易覆盖的边界分支、架构推断风险（如 WebView 音频限制）。

**疑似问题 1：<标题>**
- **可能现象**：<测试可能观察到的表现>
- **触发条件**：<需要准备的账号、数据、平台、状态或操作路径>
- **推断依据**：<来自代码路径、PRD、测试用例、平台机制或跨仓依赖的证据>
- **对应用例**：<测试用例编号/名称；没有则写"无直接对应">
- **建议验证方式**：<具体操作和观察点>

如无疑似问题，写"未发现需要额外实测确认的疑似问题"。

## 三、风险提示

- **数据库变更**：有/无，并说明依据。
- **接口变更**：有/无，并说明新增、修改、依赖方或兼容风险。
- **跨仓库依赖**：有/无，并说明前后端、H5/小程序、服务间依赖。
- **配置/开关变更**：如有，说明 Apollo、环境变量、灰度开关等影响。
- **缓存/异步/消息风险**：如有，说明对测试回归的影响。
- **权限/登录态/数据状态风险**：如有，说明需要覆盖的账号或数据状态。

## 四、建议回归场景

只列有 diff 证据影响**已有功能**的场景（旧功能被本次改动波及）。

**场景 1：<功能名>**
- **原因**：<哪个文件/函数被修改，以及为什么会影响已有功能>
- **建议操作**：<具体测试路径和观察点>

如无受影响的已有功能，写"本次改动未发现影响已有功能的回归场景"。

## 五、建议补充测试场景

列**新功能**里代码和架构风险推导出的黑盒盲区，补足黑盒测试容易遗漏的问题。
不以审查测试用例完整性为目的。

**场景 1：<功能名>**
- **补充原因**：<边界分支、状态/缓存/异步、跨端跨仓、平台环境差异、失败链路或权限登录态风险>
- **相关依据**：<PRD/用例/代码路径/架构事实>
- **建议操作**：<具体测试路径和观察点>

## 特别说明

**明确排除 / 暂缓测试**
- <PRD、用户或测试环境明确声明本次不测、暂缓、无法测试的内容>

**资料或权限不足导致未覆盖**
- <缺仓库、缺分支、缺 PRD 权限、缺测试用例、缺环境、缺数据等导致未覆盖的内容>
```

## 章节写法要求

### 架构说明（影响测试策略）

当功能涉及多仓、多端、web-view、H5、小程序、服务端或外部服务时必须写。目的不是介绍背景，而是说明实际运行位置和测试策略为什么会受影响。

### 一、发现的问题（代码直接验证）

每个问题保持简洁。问题必须能从代码直接推导出较明确的异常，不要把平台不确定性、环境不确定性或需要实机确认的问题放入本章节。

### 二、疑似问题 / 需实测确认

本章节允许更详细。"推断依据"要写清楚，帮助测试理解为什么这个点值得确认。

适合放入本章节的内容：
- 代码路径显示存在风险，但缺少实际环境或测试数据验证。
- 平台兼容、实机差异、web-view、音视频、手势、权限等问题。
- 后端字段语义、配置开关、账号状态需要确认。
- 跨仓依赖或接口契约需要实际联调确认。
- 黑盒测试主流程不容易覆盖的边界分支。

### 三、风险提示

每项都要说明和本次改动的关系，不要写成泛泛 checklist。

### 四、建议回归场景

只写 diff 影响**已有功能**的场景。重点是"这次改动可能波及了哪些旧功能"。

### 五、建议补充测试场景

不以审查测试用例为目的。重点补齐**新功能**里黑盒容易漏掉的代码和架构风险，包括边界分支、状态、缓存、异步、跨端、跨仓、平台环境差异、失败链路、权限和登录态。

### 特别说明

必须区分"明确排除 / 暂缓测试"和"资料或权限不足导致未覆盖"：
- 前者是业务或测试范围决定（PRD 写了、用户说了、测试环境不支持）
- 后者是本次扫描能力受限（缺仓库、缺分支、缺权限）

## 轻舟能力接入

报告交付后，主动询问用户哪些问题适合用轻舟能力辅助验证，不默认自动触发。

可用于辅助验证的方式：
- 需要核对接口契约时，使用轻舟 API 文档查询。
- 已有测试环境请求、错误日志或用户反馈时，使用轻舟日志查询。
- 有 traceId 或能从日志中提取 traceId 时，使用轻舟 trace 查询。
- 发布、构建或确认分支是否进入测试环境时，使用轻舟 CD 能力。

如果轻舟权限、环境或 traceId 不足，在后续验证沟通中说明，不影响提测核查报告先交付。

---

## 操作约束

- **综合访谈不可跳过**：即使用户说"直接执行""直接出报告"，也必须先完成综合访谈。"直接执行"的含义是省去信息收集的重复确认，不是跳过分析步骤。
- **质量自检结果只在对话展示**：不写入最终报告文档。
- **核心文件必须全文阅读**：不用 grep 代替，grep 只找到预设关键词，全文阅读才能发现预设之外的问题。
- **架构 × 用例必须交叉分析**：识别架构和读用例是两步，必须显式交叉——对每条用例问"这个能力在当前运行环境下有没有已知限制"。

---

## 质量标准

报告完成前自检：

1. 是否避免了"验收通过/不通过"的决策性结论。
2. 是否把确定问题和疑似问题分开。
3. 每个确定问题是否有代码位置和原因。
4. 每个疑似问题是否有详细推断依据和建议验证方式。
5. 风险提示是否和本次 diff 有关。
6. 建议回归场景是否有 diff 证据。
7. 建议补充测试场景是否聚焦黑盒容易漏的代码/架构风险。
8. 特别说明是否区分了明确排除和资料/权限不足。
