# QA 提测核查

扫描提测或 hotfix 分支相对基准分支的差异，提前发现代码问题、回归风险和测试盲区。支持完整 PRD + 测试用例模式，也支持无 PRD / 无测试用例的 hotfix 和小需求场景。

## 使用方式

```
/qa-check
```

调用后按提示提供信息即可，无需提前准备参数。

## 依赖

- **XMind 读取**：标准库即可（zipfile + xml），无需额外安装

## 执行流程

### Step 1：收集信息

**首先判断是否为复测模式**：用户提供了上一轮 qa-check 报告路径，或使用「复测」「二次提测」「再次提测」「开发修复后提测」「回归上轮问题」「re-test」等表达时，跳至 Step 1.2（复测模式），跳过下方普通提测信息收集。

向用户依次询问以下信息（一次问完，不要逐条追问）：

**硬必需输入（缺一不可）**

1. **代码仓库路径**：本地仓库的绝对路径
2. **提测分支名**：开发提测的分支（如 `feature/xxx` 或 `release/1.2.3`）
3. **基准分支**：默认 `master`，如有不同请告知

**强推荐输入（未提供不中止，但会影响分析深度）**

4. **最小变更说明**（强推荐，非硬必需）：能回答"本次改动要修什么、改什么、预期行为是什么"的任意形式说明，例如：一句话 bug 描述或需求说明、commit message、MR/PR 描述、开发或产品口头说明。如果用户未提供，流程不中止——先通过分支名、commit message、MR/PR 描述推断意图，在 Step 1.1 标记为"待 Step 2 补充推断"，Step 2 获取 diff 后继续补充，最终在报告输出前形成可追溯的最小变更口径。

**可选增强资料（越多越准，缺失不中止流程）**

5. **飞书 PRD 链接**：本次需求对应的飞书云文档链接
6. **测试用例**：支持以下任意形式：
   - XMind 文件的本地绝对路径（Claude 可直接读取 .xmind）
   - 其他格式文件的本地路径（Excel、文本等）
   - 直接将用例内容粘贴到对话中
7. **其他辅助资料**：bug 单 / 工单 / 飞书多维表链接、线上日志 / 截图 / 录屏 / 错误码、MR/PR 描述、开发或产品补充说明

PRD 和测试用例未提供时**不中止**，进入 Step 1.1 判断核查模式。

### Step 1.1：资料完整度与核查模式判定

根据用户提供的资料，判断本次核查模式，后续所有步骤的行为依模式调整。

| 模式 | 触发条件 | 允许判断 | 禁止判断 |
| --- | --- | --- | --- |
| **完整提测核查** | PRD + 测试用例 + diff | 需求符合性、代码问题、回归风险、补充测试场景 | 不输出验收通过/不通过 |
| **无用例核查** | PRD + diff，无测试用例 | 需求实现风险、代码问题、回归风险、建议补充测试场景 | 不评价用例覆盖充分/不充分 |
| **无 PRD 核查** | 有变更说明/bug 单/工单 + diff，无 PRD | 是否符合已知变更意图、代码自洽性、回归风险 | 不判断 PRD 符合性，不输出"符合 PRD" |
| **Hotfix 无文档核查** | 最小变更说明 + diff，无 PRD、无测试用例 | 修复点是否覆盖真实触发路径、是否引入回归、建议验证路径 | 不判断需求完整性，不替代正式验收，不输出验收通过/不通过 |

**判定逻辑**（按优先级）：

1. 有 PRD 且有测试用例 → **完整提测核查**
2. 有 PRD，无测试用例 → **无用例核查**
3. 无 PRD，有 bug 单 / 工单 / MR 描述 / 用户说明等变更说明 → **无 PRD 核查**
4. 无 PRD，无测试用例，仅最小变更说明或只能从 diff 推断 → **Hotfix 无文档核查**

**Hotfix 无文档核查模式：分两阶段收集最小需求口径**

**阶段一（Step 1.1 完成时）**：此时 diff 尚未获取，只记录用户已有说明（分支名、口头描述、已提供的 MR/bug 单等）。如用户未提供最小变更说明，标记为"待 Step 2 后根据 commit/diff 补充推断"，**不要在此阶段要求用户提供尚未执行 git 命令时还不存在的 diff 信息**。

**阶段二（Step 2 执行完 git log / git diff 后立即执行）**：
- 根据以下线索推断改动意图：`git log` 的 commit message、`git diff --name-only` 的文件路径、核心 diff 的函数名/变量名/接口名、新增或修改的状态字段/配置项/feature flag、日志与异常处理逻辑；
- 结合阶段一已有说明，使用 AskUserQuestion 工具一次性追问，5 个问题以内，一次问完：
  1. 这次改动要解决什么线上问题，或新增什么小能力？
  2. 用户可观察到的预期结果是什么？
  3. 影响范围是什么页面、接口、端、账号类型或业务场景？
  4. 有没有必须不能受影响的历史功能？
  5. 是否有 bug 单、线上日志、截图、录屏、错误码、MR 描述或开发说明？
- 将用户回答整理为「最小需求口径」，后续所有判断基于此口径，不擅自扩大分析范围。

### Step 1.2：复测模式（仅当 Step 1 判定为复测时执行，普通提测跳过）

**收集信息**（一次问完）：
1. 当前仓库路径和提测分支（Step 1 已填写则跳过）
2. 基准分支（默认 master）
3. 上一轮 qa-check 报告路径（优先 JSON → Markdown → HTML）
4. 开发修复说明（可选）

**执行逻辑**：

1. **读取上一轮问题清单**：优先解析 JSON 文件的 `id`、`title`、`status`、`sourceFiles` 等字段；JSON 不可用时从 Markdown 提取问题列表；仅有 HTML 时尽量提取，但必须提示：「HTML 不是推荐的结构化来源，可能无法完整恢复状态字段」。

2. **执行 Step 2** 获取当前分支 diff（与普通核查相同）。

3. **逐条判断上一轮问题状态**：
   - `Fixed`：代码证据显示相关路径已修改且修复合理，或有实测通过证据
   - `PartiallyFixed`：只修了部分路径或只覆盖部分触发条件
   - `StillOpen`：相关代码未变更，或变更未覆盖触发路径，风险仍存在
   - `CannotVerify`：缺环境、账号、数据、仓库权限或代码证据不足
   - `WontFix / KnownIssue`：用户或开发明确说明不解决 / 已知问题
   - `NeedConfirm`：需要人工实测或产品确认才能定性

4. **扫描本轮新增问题**：对本轮 diff 执行 Step 2.6 分析，检查是否引入新增问题。

5. **输出复测报告**：新文件，不覆盖上一轮报告。新问题 ID 从上一轮最大 ID + 1 开始编号。复测报告格式见 [references/report-format.md](references/report-format.md) 的「复测报告模板」。

### Step 1.5：确认项目类型与核心目录

收到仓库路径后，**在询问用户之前**先做以下探查：

```bash
ls <仓库路径>
```

再读取能反映项目类型的关键文件（只读存在的，不强求）：

| 文件 | 说明 |
|------|------|
| `package.json` | JS/TS 项目，看 dependencies 判断框架 |
| `go.mod` / `pom.xml` / `build.gradle` / `requirements.txt` / `Cargo.toml` | 后端语言 |
| `pubspec.yaml` | Flutter |
| `app.json` / `pages.json` / `manifest.json` | 小程序 |
| `AndroidManifest.xml` | Android 原生 |
| `*.xcodeproj` / `Podfile` | iOS 原生 |

探查完成后，向用户一次性确认以下两点，**等用户回复后再继续**：

> 根据仓库结构，我判断这是一个 **[项目类型]** 项目。
>
> 计划将以下目录作为核心分析范围（会全文阅读这些文件的改动）：
> `[目录1]`、`[目录2]`、`[目录3]`...
>
> 另外发现 **[是否存在跨仓库/跨层依赖]**，如：H5 通过 web-view 加载、后端调用外部服务等。
>
> 以上判断是否准确？如有不同请告知，我会按你的描述调整。

用户确认后，将「已确认的核心目录」和「已识别的架构层次」带入后续所有步骤。

### Step 2：获取代码差异

**单仓库**：对 Step 1.5 确认的仓库执行以下命令：

```bash
cd <仓库路径>
git fetch --all --prune
git diff origin/<基准分支>...origin/<提测分支> --stat
git diff origin/<基准分支>...origin/<提测分支> --name-only
git log origin/<基准分支>..origin/<提测分支> --oneline
```

如果用户提供的是本地分支而非远程分支，去掉 `origin/` 前缀。**声明分支不存在前，必须先通过 `git show-ref refs/remotes/origin/<分支名>` 或 `git ls-remote origin <分支名>` 确认远程引用，不得仅凭 fetch 前的本地 `git branch -a` 结果下结论。**

**多仓库**：对所有仓库并行执行上述命令，各自独立分析。分析完成后额外执行一次**跨仓依赖核查**：

- 检查前端/小程序调用的接口名、参数、字段，和后端改动的接口签名是否一致
- 检查后端改动的响应结构，前端是否有对应的消费侧改动
- 有一侧改动但另一侧没有同步的，标为跨仓风险，进入报告的「疑似问题/需实测确认」章节
- 检查前端触发的「写」操作（保存、完课、提交）后，后端应同步更新的状态字段是否有对应改动；前端是否依赖这些后端状态做展示或路由决策。在相关仓库、接口文档或状态机资料已提供的情况下，如果一侧改动而另一侧没有对应处理，标为跨仓风险，进入「二、疑似问题 / 需实测确认」。如果后端仓库、接口文档或状态机说明未提供，不得判断为"另一侧无对应改动"；应在「特别说明：资料或权限不足导致未覆盖」中说明缺失，并在「二、疑似问题 / 需实测确认」中列出需要确认的接口、状态字段和期望流转

最终报告按**功能模块**组织，不按仓库分章节，每个问题标注所属仓库。

**大 diff 处理策略**：获取到文件列表后，先统计变更文件数。

- **文件数 ≤ 20**：对所有文件执行完整 diff，逐一分析。
- **文件数 > 20**：分层处理：
  1. **核心文件必须读完整源码**：对落在 Step 1.5 确认的核心目录内的文件，使用 `git show origin/<提测分支>:<文件路径>` 读取完整文件内容，而不是读 diff 或用 grep 提取——grep 只能找到预设的关键词，全文阅读才能发现预设之外的问题。
  2. **非核心文件走摘要**：第三方依赖、构建配置、自动生成文件等只保留 `--stat`，不展开。
  3. **按模块分组汇报**：报告按功能模块列出变更，标注仓库来源，不逐文件罗列。

**小变更快扫模式（自动触发）**：获取文件列表后，如变更文件数 ≤ 5 且用户已提供最小变更说明，自动进入小变更快扫模式：Step 1.5 项目类型确认降级为自动识别（报告中说明，不等待用户确认）；Step 4 只追问阻断性或高价值疑点，合并为 1 次追问、≤ 2 个问题。

**旁路改动扫描（与 PRD / 变更说明做文件映射）**：

`git diff --name-only` 列出的所有文件须与 PRD / 变更说明 / bug 单 / 用户说明做映射：
- **映射上的文件**：按正常分析流程处理；
- **映射不上的文件（未说明改动 / 旁路改动）**：不直接报 bug，须进入 Step 4 疑点确认询问改动意图；用户确认为有意改动时写入「三、风险提示」，无法说明时写入「特别说明」；
- **基础设施类文件**（请求库/网络层、鉴权/权限中间件、路由配置、全局状态管理、构建配置、公共工具函数）：即使映射上了也要在「三、风险提示」中单独标注并提高回归权重。

### Step 2.6：控制流与状态依赖分析（diff 读完后必做，不可跳过）

本步骤专门补一类最易漏、且光看改动行查不出的缺陷：**改动本身没错，但新引入的路径让原本成立的前置条件失效**。典型场景是续课、恢复、跳转、免登、从缓存进入等「绕过/短路正常流程」的入口。分两部分：

**A. 新参数 / 新分支「贯穿下游」追踪**

diff 引入的每个新参数、新条件、新入口，都要追到它的**全部消费方，包括未改动的旧代码**——绝不能只看改动行。重点扫这类语义的标识：`reset / init / flag / resume / continue / restore / skip / from / mode / step / index / breakpoint`。

- **范围控制**（与「大 diff 只读核心目录」不冲突）：不是全仓通读，而是**以具体的新参数/新入口为线索定向追踪**——`git grep <参数名/函数名>` 找到消费方，再 `git show <分支>:<文件>` 读消费它的函数全文。
- 追踪时必须问：这个新取值（尤其是与默认/历史不同的取值，例如 `initReset=0`、`from=resume`）流到下游后，会不会让某段**未改动的旧逻辑**走进与首次/正常路径不同的分支？

**B.「跳过环节 → 副作用 → 是否恢复」矩阵**

识别任何**跳过 / 短路正常流程**的新路径（续课直达某题、跳转、免登快捷入口、从缓存/断点恢复等）。对它跳过的每个环节列出下表：

| 被跳过的环节 | 该环节产生的副作用（写入的状态 / 用户选择 / 过滤依据 / 高亮依据 / 进度依据 / 缓存等） | 新路径是否恢复了它 | 下游谁依赖它（含未改动代码） |
| --- | --- | --- | --- |

**判定规则**：任何一行「**未恢复 + 有下游依赖**」= 一条问题。能从代码路径确证的写入报告「一、发现的问题」，只能推断的写入「二、疑似问题」。

> 经验法则：续课/恢复类需求最常见的故障是「只保存了位置、没保存决定内容的上下文」——题目集合、单词/选项状态、选中项、高亮、进度等下游**全部依赖被跳过环节的副作用**。务必逐个下游核对，而不是只验证「位置恢复对不对」。一个被跳过环节的状态没恢复，往往会同时引发多处下游异常（如高亮错 + 进度错 + 集合变）。

**C.「下结论前先确认作用域」——避免把无害差异误报成 bug（强制）**

A/B 容易走向反面：**只看到「前端有个边攒的状态/缓存续课会丢」或「新实现比旧实现少做了一步」，就当成回归/缺陷下结论**。在写进报告前，每条疑似都必须先回答下面三问，全部确认后再定性：

1. **这个状态/字段到底有没有被消费？消费方在哪？** 用 `git grep` 把它追到所有读取方/上传方。**找不到任何消费方 = 大概率无害**，不要只凭「它会丢」就报。
2. **消费方属于哪个业务分支？**（正式课 / 试听课 / 复习课 / 某端）。常见陷阱：一段逻辑被 `if (isExperience)` / `if (isReview)` 之类门控，**它只服务某一个分支**。要把「本次需求改的分支」和「该逻辑服务的分支」对上——**两者不一致 = 与本需求无关，别报**。
3. **新旧实现差异属于哪个分支、是不是有意？** 发现「v2 比旧模板少做一步 / 多了个门控」时，先确认那一步本身服务哪个分支：若它本就是另一个分支（如试听课）专属，新实现把它门控掉往往是**有意且正确**的，不是回归。

> 反面教训：曾把「续课时某前端缓存为空」「v2 正式课完课不再调某保存接口」连续误报成正式课 bug——根因都是**没先确认该缓存/接口的消费方属于试听课、与正式课无关**就下了结论。记住：**「会丢」≠「有问题」，要先证明「有分支真的在用它」。**

**D.「保存退出触发点」完整性检查**

对涉及"保存/续课/恢复"的需求，必须枚举所有用户可能触发保存退出的操作点，而不只是验证"中途题目"场景：

| 触发保存的操作点 | 是否在本次分析中覆盖 |
| --- | --- |
| 任意中途题目保存退出 | □ |
| 最后一题/最后一道保存退出 | □ |
| 各环节边界时保存退出 | □ |
| 特殊环节（如课后检测、巧记题末尾）保存退出 | □ |

对每个触发点分别验证：续课入口能否正确定位？恢复的状态是否完整？「只在中途测过」≠「所有触发点都覆盖」。

判定规则：
- 触发点真实存在，且代码路径显示状态未保存或未恢复 → 写入「一、发现的问题」
- 触发点真实存在，但缺乏足够代码证据判断 → 写入「二、疑似问题 / 需实测确认」
- 只是测试用例未覆盖，代码无明确风险 → 写入「五、建议补充测试场景」
- 触发点在当前需求/产品路径中不存在 → 标记「不适用」，不报问题

**E. 有限循环必须有终止条件**

发现错题重入队、重试出题、循环出题等机制时，必须验证以下三点，任一不满足 = 直接写入「一、发现的问题」：

1. 重做次数是否有上限守卫（循环入队时有无 `isRepeat` 类守卫或次数上限）？
2. 最终答对是否能结束循环，而不是依赖「第一次点击」的判定导致"看着答对却仍入队"？
3. 进度计数是否将重做题纳入分母，还是锁定在初始总量导致进度死锁？

**F. 用户可感知异步操作的交互保护检查**

对保存、提交、退出、跳转、加载下一题、完课等用户可感知异步操作，检查是否存在足够的交互保护：

- 用户反馈：loading、保存中、提交中、禁用态、toast 等；
- 重复操作保护：pending guard、按钮禁用、防抖、防重复提交等；
- 失败恢复：失败提示、重试入口、状态回滚、允许重新操作等。

判定规则：
- 操作会发起远程请求或耗时任务，且代码显示没有任何用户反馈或重复操作保护 → 写入「二、疑似问题 / 需实测确认」；
- 代码无法判断真实弱网表现，但操作存在明显耗时或重复触发风险 → 写入「五、建议补充测试场景」，建议在弱网/慢接口/连续点击场景下人工验证；
- 操作为本地同步、无重复触发风险、或已有等价保护 → 不报问题。

### Step 2.5：读取测试用例（与 Step 3 并行执行）

根据用户提供的形式处理：

**XMind 文件**：使用内置脚本结构化读取，避免大文件漏读：

```bash
python3 ~/.claude/skills/qa-check/scripts/read_cases_xmind.py --xmind "<路径>"
```

脚本输出 JSON，含 `cases` 数组，每条含 `name`、`path`、`priority`、`preconditions`、`steps`、`expected` 等字段。

若脚本不可用，回退到原始解压：

```bash
unzip -l "<路径>" | grep -E "content\.(xml|json)"
unzip -p "<路径>" content.xml   # 或 content.json
```

**其他文件格式**：直接用 Read 工具读取文件内容，提取用例结构。

**直接粘贴的文本**：直接使用，无需额外处理。

**未提供**：跳过用例相关分析，在报告末尾标注。不得评价"测试用例覆盖充分/不充分"；仍可在 Step 5 基于 diff 输出「建议补充测试场景」。

### Step 3：读取 PRD（与 Step 2.5 并行执行）

**未提供 PRD 时**：跳过 PRD 读取，改用 bug 单、工单、MR/PR 描述、commit message、用户补充说明或最小变更说明作为替代规格来源。后续分析基于这些替代来源，不得输出"符合 PRD""PRD 未覆盖"之类结论。直接进入 Step 3.6 的"无 PRD 替代流程"。

**已提供 PRD 链接时**：根据飞书链接格式，按以下逻辑读取 PRD 内容：

**情况一：普通云文档**（链接包含 `/docx/` 或 `/docs/`）

直接提取 `document_id` 后调用 `docx_v1_document_rawContent`：

```
https://xxx.feishu.cn/docx/<document_id>
https://xxx.feishu.cn/docs/<document_id>
```

**情况二：Wiki 文档**（链接包含 `/wiki/`）

```
https://xxx.feishu.cn/wiki/<node_token>
```

需要两步：
1. 调用 `wiki_v2_space_getNode`，参数 `token=<node_token>`，从返回结果中取 `obj_token`（即真实的 docx token）
2. 用 `obj_token` 调用 `docx_v1_document_rawContent` 读取正文

如果 `wiki_v2_space_getNode` 返回的 `obj_type` 不是 `docx`，提示用户该 Wiki 节点类型暂不支持，请将 PRD 关键功能点直接粘贴进来。

### Step 3.5：架构 × 用例交叉分析

**仅在已提供测试用例时执行本步骤**；未提供测试用例时跳过，直接进入 Step 3.6。

Step 2（架构识别）和 Step 2.5（测试用例）都完成后，执行一次交叉检查：

**对每一条测试用例，判断它依赖的能力在当前运行环境下是否存在已知限制。**

重点关注以下能力类型，这些在非标准运行环境中最容易出现平台约束：

- 音视频自动播放（autoplay）
- 系统返回 / 导航拦截（back button、手势返回）
- 文件上传 / 下载
- 剪贴板读写
- 推送通知 / 后台保活
- 摄像头 / 麦克风 / 定位等系统权限
- 本地存储（容量限制、隔离策略）
- 网络请求（跨域、证书限制）

**操作方式**：逐条扫描测试用例标题和步骤，凡是涉及上述能力的，立即对照已识别的运行环境（web-view、小程序原生、RN 等），判断该环境对该能力是否有已知限制。有则记录为「架构推断风险」，进入最终报告的「疑似问题 / 需实测确认」章节。

此步骤不依赖 diff，不需要读 H5 代码，只需要：用例说明了什么 + 当前环境的已知约束。

### Step 3.6：提取 PRD 测试重点

**有 PRD 时**：读完 PRD 后，立即提取以下四类内容，作为后续分析的优先检查清单：

1. **PRD 中明确标注为"重点"/"测试重点"的功能**：逐条列出，分析时必须对照 diff 逐一核查，报告中必须逐条说明是否覆盖；若功能逻辑在 H5 仓库而非本仓库，也要在报告中明确标注
2. **PRD 中标注为"暂不支持"/"测试环境无法测试"的功能**：在报告中单独列出，提示测试阶段跳过
3. **质量 / 脏数据隐患信号**（静态读代码常查不出，必须转成测试场景）：
   - PRD 里明说的内容/数据质量问题（如「拓展词质量问题：有问题的拓展词」「部分数据可能缺失/异常」等）；
   - 代码里暴露的「已知脏数据」线索：黑名单、过滤注释（如「过滤缩写词等」）、特殊 case 兜底、`try/catch` 吞掉的异常分支。
   - 凡命中，转成一条**数据质量黑盒场景**写入报告「五、建议补充测试场景」——即使代码逻辑层面查不出对错（出不出问题取决于数据），也要提示测试用真实数据验证内容质量（如是否混入短语/缩写/无效词/空值）。

4. **PRD 中的计数/判定触发规则与资源降级策略**：
   - 提取所有「X 操作 → 状态/计数变化」的明确条件（如"只有答对才 +1""第一次点击决定对错""最终结果才判定"）；
   - 提取资源集合不足、为空、过滤后不足、接口部分失败时的补足、降级、跳过或兜底策略（如"不足时从下一批次/分页/单元补充""无法获取时展示空状态"）；
   - 后续代码扫描时，对每条规则寻找对应实现并按证据强度分类：
     - 代码实现与 PRD 明确冲突 → 写入「一、发现的问题」或「二、疑似问题」，取决于证据强度；
     - 当前仓库应负责该规则，但找不到实现 → 写入「二、疑似问题 / 需实测确认」；
     - 规则可能由其他仓库、配置、数据或后端状态机负责，且资料未提供 → 写入「特别说明：资料或权限不足导致未覆盖」，必要时补充至「二、疑似问题」；
     - PRD 未明确具体期望 → 标记"需确认产品预期"，不直接判定为 bug；
     - 本次明确排除或暂不支持 → 写入「明确排除 / 暂缓测试」，不计入问题。

**无 PRD 时**：从以下来源提取重点验证点，替代 PRD 提取流程：
- 最小变更说明 / 最小需求口径中的预期行为和影响范围；
- bug 单 / 工单 / 用户说明中的触发路径和期望结果；
- commit message 和 MR/PR 描述中的功能意图；
- 代码里暴露的数据质量线索（同上述第3类，仍需转成测试场景）。
不得输出"符合 PRD""PRD 要求已实现"；改用"符合变更说明意图"表述。

此步骤在 Step 3 完成后立即执行，不等待 Step 2.5。

### Step 4：综合访谈

完成 Step 2、Step 2.6 之后，**必须执行本步骤，不可跳过**。即使用户在初始 prompt 中说了「直接执行」「直接出报告」等字样，也必须先完成本步骤。Step 2.5、Step 3、Step 3.5、Step 3.6 可能因资料缺失被跳过，不影响本步骤执行。

**降级模式下的追问重点**：
- **完整提测核查**：按原逻辑执行范围对齐、架构追问、疑点确认；
- **无用例核查**：跳过用例相关追问，其余保持原逻辑；
- **无 PRD 核查**：围绕变更意图、影响范围、预期行为、排除范围追问；
- **Hotfix 无文档核查**：最小需求口径已在 Step 1.1 收集，本步骤只追问架构疑点和排除范围，问题控制在 3 个以内，避免重复追问；
- **小变更快扫模式**：只追问阻断性疑点（旁路改动意图、跨仓缺口），合并为 1 次追问、≤ 2 个问题，跳过低价值确认。

使用 **AskUserQuestion 工具**将以下三类问题结构化呈现，一次问完，问题总数控制在 5 题以内，超出时合并低优先级问题。用多选选项减少用户手打文字的负担。

**一、范围对齐（每次必问）**

将以下 4 个问题整合为 AskUserQuestion 的选项，让用户勾选：
- 审查模式：全面扫描 / 有明确重点区域（注明是哪里）
- 已在其他平台或阶段验证过的功能，本次可跳过
- 测试环境限制或不在本次需求内的功能
- 已有顾虑或希望重点关注的地方

**二、架构追问（根据 Step 1.5 + Step 3.5 的发现按需触发，不相关的跳过）**

| 发现的特征 | 追问内容 |
|-----------|---------|
| 存在 web-view 加载 H5 | H5 仓库是否也在本次提测范围？ |
| 多平台构建（微信/抖音/Android/iOS） | 哪些平台需要覆盖，哪些暂不测试？ |
| 存在数据库 migration | 是否已通知 DBA 审核？是否需要回滚方案？ |
| 新增或修改对外 API | 调用方是否已同步？是否需要版本兼容？ |
| 涉及第三方服务 | 测试环境的第三方服务是否可用？ |
| 存在配置开关 / Feature Flag | 测试环境开关状态是否已确认？ |

**三、疑点确认（根据分析结果和核查模式按需触发）**

**有 PRD 时**：
1. **代码中有改动、但 PRD 未提及的逻辑**：逐条描述，询问是否属于本次需求范围
2. **PRD 提及、但 diff 中未找到对应改动的功能点**：询问是否在其他仓库或已提前上线

**无 PRD / 无 PRD 核查模式**：
1. **代码中有改动、但变更说明 / bug 单 / 工单 / MR 描述 / 用户补充说明未提及的逻辑**：逐条描述，询问是否属于本次改动范围
2. **变更说明中提及、但 diff 中未找到对应改动的修复点或功能点**：询问是否在其他仓库或已提前上线

**Hotfix 无文档核查模式**（围绕最小需求口径）：
1. **代码中新增或修改、但最小需求口径未提及的逻辑**：逐条描述，询问是否属于本次修复范围
2. **最小需求口径中提及、但 diff 未体现的修复点**：询问是否在其他仓库、配置或后端已处理

---

用户回答后，将所有答案作为最终约束带入 Step 5：重点区域深挖、排除范围略过、已有顾虑优先核查、疑点按确认结果处理。

### Step 4.5：质量自检（报告输出前）

报告内容生成完毕后，对以下标准逐一自检。**自检结果只在对话中告知用户，不写入最终文档。** 有不达标项则修正后再输出：

1. 是否避免了"验收通过/不通过"的决策性结论
2. 是否把确定问题和疑似问题分开
3. 每个确定问题是否有代码位置和原因
4. 每个疑似问题是否有详细推断依据和建议验证方式
5. 风险提示是否和本次 diff 有关
6. 建议回归场景是否有 diff 证据
7. 建议补充测试场景是否聚焦黑盒容易漏的代码/架构风险
8. 特别说明是否区分了明确排除和资料/权限不足
9. （Step 2.6）是否对每个 resume/恢复/跳转/免登等绕过正常流程的入口，做了「跳过环节 × 副作用 × 是否恢复 × 下游依赖」矩阵？矩阵里「未恢复+有下游依赖」的行是否都落成了问题？
10. （Step 2.6）diff 引入的新参数/新入口，是否都追到了所有下游消费方（含未改动的旧代码），而不是只看改动行？
11. （Step 3.6）PRD 明示的质量隐患、代码里黑名单/过滤注释暴露的已知脏数据，是否都转成了数据质量测试场景？
12. （Step 2.6-C）每条疑似/确定问题，是否都已确认「相关状态/字段/接口有真实消费方」+「消费方分支与本次需求改动的分支一致」？凡是「会丢/少做一步」但没找到正式消费方、或消费方属于本需求范围外分支（如试听课）的，必须排除，不得报为问题。
13. （Step 2.6-D）涉及保存/续课/恢复的需求，是否枚举了全部四类触发保存退出的操作点，并按落报规则区分了确定问题、疑似问题、测试场景、不适用四档？
14. （Step 3.6 第4类）PRD 中的计数/判定触发规则和资源降级策略，是否全部提取并逐条找到了对应代码实现？是否区分了"当前仓库应负责但缺实现"和"可能在其他仓库/资料未提供"？
15. （Step 2.6-F）涉及用户可感知异步操作的需求，是否检查了是否有足够的交互保护？是否按"有代码证据/无法从代码判断/操作无风险"三档落报？
16. （报告章节）报告章节是否与核查模式匹配？非 Hotfix 无文档核查模式下，是否没有输出「最小需求口径」章节，也没有保留任何模板注释或"删除本节"之类的编辑提示？

### Step 5：生成核查报告并写入文档

综合 git diff、PRD 内容、测试用例及用户确认结果生成报告并写入文档。**完整的 Markdown 报告模板、JSON 问题台账 schema、HTML 生成规则、条件章节输出规则、复测报告模板，见 [references/report-format.md](references/report-format.md)。**

**输出目录结构（按需求组织，不平铺到桌面）**：

根路径：`~/Desktop/qa-check-reports/`

每次运行在以下路径写入文件：

```
~/Desktop/qa-check-reports/<需求名称>/<日期>-<分支名>-<shortCommit>/
  draft/              # 初版报告（静态扫描，默认输出位置）
    qa-check-draft.md
    qa-check-draft.json
    qa-check-draft.html
  final/              # 终版报告（用户提供测试入口并完成实测后生成）
    qa-check-final.md
    qa-check-final.json
    qa-check-final.html
  recheck/            # 复测报告（复测模式输出）
    qa-check-recheck-<日期>.md
    qa-check-recheck-<日期>.json
    qa-check-recheck-<日期>.html
  artifacts/          # 证据文件（截图/录屏/日志/接口响应）
    videos/
    screenshots/
    logs/
```

**路径分段说明**：

- `<需求名称>`：优先取用户提供的需求说明或 PRD 标题，其次取分支名去掉 `feature/`/`hotfix/` 前缀后的可读部分，不超过 20 个字符，仅用字母/数字/汉字/连字符，不含空格；
- `<日期>`：`YYYYMMDD` 格式；
- `<分支名>`：分支名中 `/` 替换为 `-`；
- `<shortCommit>`：`git rev-parse --short HEAD` 取前 7 位；
- 以上路径在输出前由 `mkdir -p` 自动创建，无需用户手动创建。

**文件生成规则**：Markdown / JSON / HTML 三类文件必须同时生成，写入相同子目录。Markdown 是人工可读/可编辑的主报告；JSON 是结构化台账，用于复测、状态追踪和重新生成 HTML；HTML 是展示层，不是编辑源，不建议手工维护，如需更新状态应重新运行 qa-check 复测模式读取 JSON 生成新版本。

**报告阶段规则（三阶段，不可混用）**：

| 阶段 | 目录 | 触发条件 | 允许输出 | 禁止输出 |
| --- | --- | --- | --- | --- |
| `draft` | `draft/` | 静态扫描完成，无需真实测试环境 | 代码证据、理论风险、疑似问题、建议测试场景、资料缺口 | "已实测复现"、实测证据文件路径 |
| `final` | `final/` | 用户提供测试入口和账号数据后，完成真实验证 | 实测状态、证据文件路径、更新后的问题状态 | 未经真实验证的"已修复"结论 |
| `recheck` | `recheck/` | 开发修复后再次提测，读取上一轮 JSON | 复测状态、新增问题、证据更新 | 复用旧 ID 给新问题 |

**飞书写入（可选）**：如飞书可用，调用 `docx_builtin_import`，`markdown` 为报告完整内容，`file_name` 为 `QA核查报告-<需求名称>-<今日日期>`，`useUAT: true`，成功后告知飞书链接。飞书写入成功或失败均不影响本地三类文件的生成。

**产物位置回执（报告生成后必须在最终回复中醒目输出，HTML 放第一位）**：

```
📦 产物位置
  HTML（可直接打开）：<绝对路径>/draft/qa-check-draft.html
  Markdown：         <绝对路径>/draft/qa-check-draft.md
  JSON 台账：        <绝对路径>/draft/qa-check-draft.json
```

生成后检查三类文件存在且大小 > 0，写入失败立即告知用户。Draft 报告生成后必须提示：「提供测试 URL / 账号 / 数据后可生成 final 终版报告」。

**实测前置资料收集规则**：

在任何实测、截图、录屏操作前，必须先判断用户是否已提供测试资料。**Claude 不得使用代码里的域名、历史文档中的链接、线上地址、旧测试环境地址、猜测出来的入口**作为测试入口。如果缺少资料，先输出「实测资料需求清单」，一次性向用户确认，按平台/模块聚合，控制在 5 个问题以内。可询问的资料类型：Web/H5 测试环境 URL；App 测试包/安装方式/DeepLink/目标页面路径；小程序测试版入口/二维码/页面路径；测试账号/角色/登录方式/验证码处理；测试数据或构造方式；feature flag/灰度开关/后端接口 base URL/token；特殊用户状态或历史数据状态。用户提供资料前，不得声明"已实测复现"，只能输出理论复现路径和代码证据。

**录屏证据采集规则**：

录屏不是每个问题都必须做。只有以下情况才优先尝试：用户明确要求；P0/P1 高风险问题；上轮未关闭问题（StillOpen / PartiallyFixed）；开发声称已修复但代码证据不足；静态扫描无法完全定性但具备明确用户路径。

- 只有真实运行并观察到异常，才能标记 `evidenceLevel: L2-RuntimeVerified`；
- 只有真实生成了录屏/截图/日志/接口响应文件，才能写入 `evidenceFiles`；
- **禁止伪造截图、录屏、日志、接口响应、命令输出**；
- 没有真实证据文件时，`evidenceFiles` 必须为空数组 `[]`。

**落报规则（内部执行规则，不写入最终报告）**：

- 有明确代码证据，且能说明用户可感知异常 → 写入「一、发现的问题」，标注 `issueOrigin`
- 有代码风险、有明确用户路径或 PRD/用例支撑，但缺实测证据 → 写入「二、疑似问题 / 需实测确认」
- 低概率理论边界、缺乏明确业务预期的视觉差异、纯数据构造建议 → 写入「五、建议补充测试场景」，不包装成疑似 bug
- 旁路改动（未说明改动） → 进入 Step 4 疑点确认 + 写入「三、风险提示」，不直接报 bug
- UI/样式/打印/PDF/视觉类需求，未拿到测试入口或截图/PDF → 写入「五、建议补充测试场景」（输出视觉验证矩阵和打印/导出风险点），不写成确定 bug
- 因缺少 PRD、测试用例、接口文档、后端仓库、线上数据导致无法判断 → 写入「特别说明：资料或权限不足导致未覆盖」

**禁止输出的结论（内部规则，不写入最终报告）**：
- 无 PRD 时，禁止输出"符合 PRD""需求已完整实现"
- 无测试用例时，禁止评价"测试用例覆盖充分/不充分"
- Hotfix 无文档核查模式下，禁止输出"验收通过/不通过"
- 资料不足时，不能把纯推测写成确定问题
- 不得把低概率理论边界或纯数据构造建议写成疑似 bug
